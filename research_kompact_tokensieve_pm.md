# Kompact / TokenSieve 集成评估：产品经理视角

> 在给出技术判断之前，先明确：我们要解决什么问题、用什么维度衡量成功、以及从哪些角度做取舍。

---

## 一、产品定位回顾

当前项目 `llama.cpp`（实际为本地 LLM 推理编排层）是一个 Anthropic-compatible API 代理：

```
Claude Code → anthropic_proxy.py:4000 → llama-server/rapid-mlx:8081 → GGUF/MLX 模型
```

**核心约束**：
- 运行设备：MacBook Pro M5 Pro，48 GB unified memory
- 主力模型：Qwen3.6-35B-A3B via Rapid-MLX（偶尔 Qwen3.6-27B-MTP via llama-server）
- 使用场景：长时 agentic coding 会话，Claude Code 作为客户端
- 成本结构：本地运行无按 token 计费，但受限于内存、上下文窗口、推理延迟

**因此，这里的"成本"主要不是 API 费用，而是**：
1. 上下文窗口是否被快速撑满
2. KV cache / 内存是否会触发 OOM 或性能下降
3. 模型是否因噪声上下文而降低任务完成质量
4. 长会话是否会陷入 re-read / wasted call 等死循环

---

## 二、产品目标（Goal）

### 2.1 北极星目标

> **在保持或提升 Agent 任务完成率的前提下，降低长会话的上下文压力，延缓上下文窗口耗尽，减少模型因噪声产生的无效动作。**

### 2.2 二级目标

| 目标 | 为什么重要 | 衡量方式 |
|---|---|---|
| G1: 降低单轮请求 token 量 | 直接缓解 KV cache 和上下文压力 | 每轮请求 input_chars / 估算 tokens |
| G2: 提高 prefix cache 命中率 | Rapid-MLX/llama-server 可复用 KV，降低重复计算 | 启动后相同前缀请求的 latency / tokens |
| G3: 减少长会话中的无效循环 | 降低 re-read、wasted call、blocker 类错误 | `logs/proxy_metrics.jsonl` 中 quality_flags 与 wasted 计数 |
| G4: 保持或提升任务完成率 | 压缩不能破坏模型理解 | 端到端任务成功数 / 人工抽检 |
| G5: 保持可观测与可调试 | 压缩后仍需知道"发生了什么" | metrics 日志完整性、错误可追踪 |
| G6: 控制实现与维护成本 | 避免引入过重依赖或复杂架构 | 新增代码量、新增依赖数、测试覆盖 |

### 2.3 非目标

- **不追求最大压缩率**：TokenSieve 对 AWS CLI 可达 66%，但我们的输入以代码文件、对话历史、tool result 为主，不能简单套用。
- **不改变客户端交互模型**：Claude Code 仍通过 `127.0.0.1:4000` 连接，不引入额外配置步骤。
- **不做跨会话持久记忆**：本次评估聚焦单次会话内的上下文优化，不涉及长期记忆层。

## 三、评估维度与判断标准

### 3.1 维度一：压缩收益（Impact）

**问题**：能省多少 token / 内存 / 延迟？

| 子指标 | 定义 | 合格线 | 优秀线 |
|---|---|---|---|
| 平均请求压缩率 | `(原 token - 压缩后 token) / 原 token` | ≥ 15% | ≥ 35% |
| 长会话尾部衰减 | 会话 30 分钟后单轮请求是否仍可控 | 增长 ≤ 2x | 增长 ≤ 1.5x |
| prefix cache 命中提升 | 相同 system prompt 连续请求的命中比例 | 有提升 | 明显提升 |
| 内存压力缓解 | 相同任务下峰值内存使用 | 持平或下降 | 明显下降 |

**判断角度**：
- 不能只看单次压缩率，要看**长时会话中的累积收益**。
- 对本地后端，token 节省最终转化为 KV cache 节省，但前提是前缀稳定。

### 3.2 维度二：质量保持（Quality）

**问题**：压缩后模型还能不能正确理解上下文、完成任务？

| 子指标 | 定义 | 合格线 | 优秀线 |
|---|---|---|---|
| 任务完成率 | 端到端 benchmark / 人工抽检成功率 | 下降 ≤ 5% | 持平或提升 |
| re-read / wasted 错误增长 | `Wasted call` 类错误数量 | 不增加 | 减少 |
| 关键信息丢失 | Read 文件内容、错误信息、系统指令是否完整 | 零丢失 | 更精简但完整 |
| 模型困惑度 | 压缩后模型输出是否出现更多"我不确定"类表述 | 不明显增加 | 无增加 |

**判断角度**：
- 压缩导致的质量损失必须**可量化、可回滚**。
- 对 coding agent，**代码文件内容和错误信息是高压线**，不能丢。

### 3.3 维度三：稳定性与风险（Risk）

**问题**：引入后会不会让系统更脆弱？

| 子指标 | 定义 | 合格线 | 优秀线 |
|---|---|---|---|
| 新增依赖 | 新增第三方库 / 外部服务 | 0 或纯标准库 | 少量、成熟、易安装 |
| 失败回退 | 压缩失败时能否透传原始请求 | 必须有 | 自动降级 + 日志 |
| 并发安全 | 多请求同时处理是否安全 | 无竞态 | 已验证 |
| 配置复杂度 | 新增 env vars / 开关数量 | ≤ 3 个 | 1 个总开关 + 可选微调 |
| 与现有特性冲突 | 是否破坏 tool filter、smart truncation、blocker 等 | 不冲突 | 协同增强 |

**判断角度**：
- 本地后端已经够脆弱（OOM、超时、TurboQuant 问题），不应再引入新的不稳定因素。
- 所有压缩开关应默认关闭，先灰度验证。

### 3.4 维度四：可观测性（Observability）

**问题**：压缩后出了问题，能否快速定位？

| 子指标 | 定义 | 合格线 | 优秀线 |
|---|---|---|---|
| per-transform 指标 | 每个压缩步骤的节省 token 数 | 有统计 | 有详细 breakdown |
| 原始内容可恢复 | 是否保留原始请求快照 | 保留关键字段 | 完整保留 |
| 日志可读性 | 压缩后提示是否仍能让人类理解 | 基本可读 | 高度可读 |
| 告警能力 | 压缩率异常、质量下降是否可检测 | 可手动检测 | 可自动阈值告警 |

**判断角度**：
- 当前 proxy 已有 `proxy_metrics.jsonl` 和 `proxy_requests.jsonl`，新能力必须接入这套体系。
- 压缩不能成为黑盒。

### 3.5 维度五：工程成本（Cost）

**问题**：投入产出比如何？

| 子指标 | 定义 | 合格线 | 优秀线 |
|---|---|---|---|
| 实现工作量 | 人日 | ≤ 3 天 | ≤ 1 天 |
| 测试覆盖 | 单元 / 集成测试是否足够 | 覆盖核心路径 | 覆盖边界场景 |
| 维护成本 | 后续调优、bug 修复负担 | 低 | 极低 |
| 性能开销 | 压缩本身耗时 | < 10ms/请求 | < 1ms/请求 |

**判断角度**：
- 对单人维护的项目，应优先选择**改动小、逻辑清晰、易回滚**的方案。
- 避免引入新的运行时（如 Kompact 的 FastAPI 独立服务）。

## 四、决策角度：如何做取舍

### 4.1 角度一：问题驱动 vs 方案驱动

**问题驱动**：先诊断当前 proxy 在长会话中的真实瓶颈，再决定引入哪种压缩。

建议查看 `logs/proxy_metrics.jsonl` 回答：
- 当前单轮请求平均多少 token？
- 增长到上限（180K chars）平均需要多少轮？
- 主要增长来源是 tool_result、tool definitions、还是 assistant thinking？
- wasted call / loop / blocker 类错误频率如何？

**如果 tool_result 占大头**：优先 TokenSieve 式 JSON/日志结构化压缩。
**如果 tool definitions 占大头**：优先 Kompact Schema Optimizer。
**如果 system prompt 动态值导致 cache 不命中**：优先 Cache Aligner。

### 4.2 角度二：局部优化 vs 系统优化

| 类型 | 例子 | 适用场景 |
|---|---|---|
| 局部优化 | 压缩单个 tool_result 的内容 | 见效快、风险低、可独立 A/B |
| 系统优化 | 重写整个 pipeline 引入 Kompact 全部 9 个 transform | 收益大但风险高、改动面广 |

**建议**：先做局部优化，验证有效后再考虑系统化集成。

### 4.3 角度三：保守 vs 激进

| 策略 | 特点 | 适用 |
|---|---|---|
| 保守 | 默认关闭，仅对明确场景开启；保留原始内容透传 | 当前阶段更推荐 |
| 激进 | 默认开启，最大化压缩率 | 需要大量验证后才能考虑 |

考虑到本地后端的稳定性和当前项目的单人维护属性，**推荐保守策略**。

### 4.4 角度四：单次收益 vs 长期收益

| 收益类型 | 例子 |
|---|---|
| 单次收益 | 某条 AWS CLI 输出从 14K token 降到 8K |
| 长期收益 | Cache Aligner 让每次请求 prefix cache 都命中，累积省大量重复 KV |

**Cache Aligner 是典型的长期收益**，单次请求节省不明显，但长会话累积收益巨大。
**JSON 压缩是典型的单次收益**，立竿见影。

### 4.5 角度五：可逆性

> **任何压缩策略都必须是可逆的或至少是可回滚的。**

- 新增功能必须有独立开关。
- 保留原始请求副本（至少在 debug 模式下）。
- 能在配置中一键回到当前行为。

## 五、候选方案优先级初判

基于以上维度，对三个最可能的落地方案做初步优先级排序。

### 5.1 方案 A：Cache Aligner（最高优先级候选）

**目标对应**：G1, G2, G3, G5, G6
**实现**：在 proxy 中对 system prompt 和前 1–2 条 user 消息做 UUID/时间戳/用户路径占位符化。

| 维度 | 评分 | 理由 |
|---|---|---|
| 压缩收益 | ★★★☆☆ | 单次 token 节省有限，但 prefix cache 命中收益长期显著 |
| 质量保持 | ★★★★★ | 无损替换，模型仍看到占位符 |
| 稳定性 | ★★★★★ | 纯文本替换，无新增依赖 |
| 可观测性 | ★★★★☆ | 需记录替换的动态值映射 |
| 工程成本 | ★★★★★ | 1–2 天可完成 |

**风险**：占位符变化后模型是否仍能正确引用？需验证 Claude Code 对 `{UUID_0}` 这类占位符的理解能力。

### 5.2 方案 B：Tool Result JSON/日志结构化压缩（高优先级候选）

**目标对应**：G1, G3, G5
**实现**：对 tool_result 中的 JSON 字符串应用 sieve（null/空值/base64 占位）+ deduper（重复标量去重）+ 可选 minify/Schema-YAML。

| 维度 | 评分 | 理由 |
|---|---|---|
| 压缩收益 | ★★★★☆ | AWS/CLI 类输出可省 30–50%，代码类输出收益较低 |
| 质量保持 | ★★★★☆ | 需避免误删代码文件中的关键标量 |
| 稳定性 | ★★★★☆ | 纯本地处理，失败可透传原 JSON |
| 可观测性 | ★★★★☆ | 记录压缩前后 token 数 |
| 工程成本 | ★★★★☆ | 2–3 天可完成核心逻辑 |

**风险**：
- 模型依赖 JSON 格式理解结构化数据，Schema-YAML/PVFN 需要适应期。
- 对 Read 代码文件的结果不应启用，避免破坏代码。

### 5.3 方案 C：TF-IDF Schema Optimizer（中优先级候选）

**目标对应**：G1, G2（tool defs 减少后缓存更稳定）
**实现**：替换或增强当前 `_filter_tools`，用 TF-IDF 按 query 动态选择 tool。

| 维度 | 评分 | 理由 |
|---|---|---|
| 压缩收益 | ★★★☆☆ | 仅在 tool 数量 > 20 时明显 |
| 质量保持 | ★★★☆☆ | 动态选择可能漏掉后续需要的 tool |
| 稳定性 | ★★★☆☆ | 需要更多测试确保不误删关键 tool |
| 可观测性 | ★★★★☆ | 可记录 filtered_out 列表 |
| 工程成本 | ★★★☆☆ | 3–5 天，需设计打分逻辑与回退策略 |

**风险**：
- 当前 `_filter_tools` 已通过白名单 + 最近使用实现稳定压缩，替换为动态 TF-IDF 的收益/风险比不够高。
- 建议作为**增强项**而非替换项：先保留白名单，再对非白名单 tool 做相关性排序。

### 5.4 方案 D：完整引入 Kompact 作为独立服务（最低优先级）

**目标对应**：G1, G2
**实现**：在 anthropic_proxy.py 前再挂一层 Kompact 代理，或替换当前代理。

| 维度 | 评分 | 理由 |
|---|---|---|
| 压缩收益 | ★★★★☆ | 综合能力较强 |
| 质量保持 | ★★★☆☆ | 8+ transform 叠加，难以定位问题 |
| 稳定性 | ★★☆☆☆ | 新增服务、新增依赖（FastAPI、OpenTelemetry 等） |
| 可观测性 | ★★★★☆ | Kompact 自带 dashboard |
| 工程成本 | ★★☆☆☆ | 架构改动大，与当前 proxy 能力重复 |

**结论**：不推荐直接引入独立 Kompact 服务。应**借鉴其具体 transform 实现**，将其能力以函数形式嵌入当前 proxy。

## 六、决策建议（前置结论）

在未完成具体实验前，基于产品经理视角的初步建议：

1. **先做 Cache Aligner PoC**（方案 A）
   - 收益长期、风险最低、改动最小。
   - 可直接提升 prefix cache 命中率，对本地 KV cache 压力有累积缓解。

2. **并行验证 Tool Result JSON 压缩**（方案 B）
   - 针对 Bash/Read 返回的 JSON、日志类 tool_result。
   - 默认关闭，通过 env 开关灰度，只作用于非代码文件结果。

3. **暂缓 Schema Optimizer 重构**（方案 C）
   - 当前 `_filter_tools` 已足够稳定，动态 TF-IDF 收益不确定。
   - 可作为后续增强，但不作为优先项。

4. **拒绝完整替换为 Kompact/TokenSieve 独立服务**（方案 D）
   - 与当前架构冲突，维护成本过高。

## 七、验证计划建议

在最终决策前，建议跑以下实验：

| 实验 | 目的 | 所需数据 |
|---|---|---|
| E1: 分析 `logs/proxy_metrics.jsonl` | 确定当前主要 token 增长来源 | 最近 5–10 次长会话日志 |
| E2: Cache Aligner A/B | 对比开关前后 prefix cache 命中与 latency | 相同 system prompt 重复请求 10 次 |
| E3: JSON 压缩 A/B | 对比真实 tool_result 压缩前后的 token 与任务完成率 | 20–50 条 Bash/Read JSON 输出 |
| E4: 端到端长会话测试 | 验证 30 分钟 agentic 会话是否更稳定 | 2–3 个标准 coding 任务 |
