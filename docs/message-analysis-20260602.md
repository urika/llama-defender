## 十三、根因总结

### 13.1 五个核心根因

1. **线性膨胀不可避**
   - 每轮工具循环 +~2,300 chars，85 轮 = 195K messages
   - 这是 agentic 编程的固有特性

2. **结构开销固化**
   - 即使内容被清空，170+ 条 message 对象的 JSON 包装仍占 ~30K chars
   - 无法通过清理减少

3. **工具定义固定开销**
   - 92K chars / ~26K tokens 的工具定义是每次请求的基础税
   - 从 42→27 工具节省了 ~25K chars，但仍是主要开销

4. **代理清理效果显著但有限**
   - 清理释放了 100K chars（33%），将 85K tokens 降到 56K
   - 但无法解决结构层面的线性膨胀

5. **后端物理极限**
   - 48GB RAM + 35B 模型 = 56K tokens 已是极限
   - 任何进一步增长都会突破内存阈值，导致 forced cache clear 和 prefix cache 失效

### 16.2 Agent 行为模式诊断

#### 16.2.1 「零思考」模式 — 最严重问题

**现象**：85 轮循环中，Assistant 从未输出过不含 `tool_use` 的纯文本回复。

**影响**：
- 模型没有「暂停-思考-规划」的步骤，每一步直接进入工具调用
- 无法批量规划任务，只能「走一步看一步」
- 用户看不到任何中间推理过程，透明度为零
- 模型可能在「盲目」调用工具，而非基于全局理解行动

**根因**：
- System prompt 可能过度强调「立即行动」
- 或模型量化后（4-bit）推理能力下降，倾向于安全地调用工具而非自主回答
- Claude Code 的 prompt 设计可能默认「能工具就不说话」

#### 16.2.2 「Read-Bash 跑步机」— 低效循环

**现象**：Bash 42.9%、Read 42.9%，两者交替出现，60% 为重复调用。

**典型循环**：
```
Round N:   Read(file1) → 获取内容
Round N+1: Bash(grep/edit) → 处理
Round N+2: Read(file2) → 验证
Round N+3: Bash(grep/edit) → 再处理
...
```

**问题**：
- 每次只读一个文件、执行一个命令
- 没有「批量读取」或「批量执行」的规划
- 大量往返开销（每轮 +2,300 chars）换取微小进展

**理想模式**：
```
Round 1: 规划 → 列出需要读取的10个文件
Round 2: 批量 Read(file1-10)
Round 3: 分析 → 确定需要修改的3处
Round 4: 批量 Bash(edit1, edit2, edit3)
Round 5: 验证 → 批量 Read 验证
```

#### 16.2.3 「失忆症」— 上下文利用率极低

**现象**：
- PROXY_TOOL_KEEP=2 清理了 83-86 个旧 tool_result
- 这些旧结果被替换为占位符 `[cleared to save context: X chars]`
- 模型实际看到的完整内容只有**最近的 2 个**

**影响**：
- 模型对 83 轮之前的工作几乎没有记忆
- 170 条消息的结构开销（30K chars）白白占用 token，却不提供任何信息
- 模型被迫在「信息匮乏」的状态下做决策
- 可能导致重复修改同一文件、忽略早期约束、做出矛盾决策

**悖论**：Agent 看似有 85 轮「经验」，实际上只有 2 轮的「记忆」。

#### 16.2.4 「膨胀-重置」恶性循环

**现象**：8 次 compact，每次 compact 后迅速恢复大上下文。

```
膨胀 ──→ 197K ──→ Compact ──→ 6K ──→ 恢复 ──→ 193K ──→ Compact ──→ 8K ──→ ...
```

**问题**：
- Agent 无法维持「轻量」工作状态
- 每次 compact 后需要重新加载工具、重新建立上下文
- 用户被迫在「等待 compact 完成」和「等待膨胀崩溃」之间循环
- 8 次 compact 意味着约 **8 次工作中断**，严重影响连续性

#### 16.2.5 「工具配置恐慌」— 不稳定的工作环境

**现象**：27 次工具配置切换，平均每 4.6 个请求切换一次。

**问题**：
- Agent 可用的工具在 0/27/42/44 之间反复横跳
- 模型刚适应一组工具，工具集就变了
- 可能导致模型「不知道该用什么」，反复试错
- 44 tools 的小请求（6K-24K）可能是 Claude Code 内部在做工具发现/探测

### 16.3 Agent 认知质量评估

#### 16.3.1 注意力稀释

- 56K tokens 输入，其中 30K 是「空壳」消息（已清理的占位符）
- 模型需要将注意力分散到 170 条消息上
- 研究表明，长上下文中模型对中间位置的注意力显著下降
- **结论**：模型可能「看」到了 170 条消息，但只「记住」了最近的 2-3 条

#### 16.3.2 决策疲劳

- 85 轮连续工具调用，无休息/总结
- 每轮增加 2,300 chars 的认知负担
- 模型在后期（req 51-75 膨胀 70K）可能出现「决策质量下降」
- 类似人类的「决策疲劳」：后期更可能做出次优或重复决策

#### 16.3.3 幻觉风险

- 当 83 个 tool_result 被清理为占位符时，模型看不到实际内容
- 但模型可能基于**自己的假设**填充缺失信息
- 这可能导致：
  - 假设文件内容未变（实际已修改）
  - 假设命令输出与之前相同（实际不同）
  - 基于幻觉做出错误修改

### 16.6 Agent 工作效果核心结论

当前会话呈现 **「高频率、低产出、强循环、弱记忆」** 的典型症状：

| 症状 | 数据支撑 | 影响 |
|------|----------|------|
| **零思考** | 0% 纯文本回复 | 缺乏规划，盲目工具调用 |
| **跑步机效应** | Read-Bash 42.9% 各半，60% 重复调用 | 大量往返，产出微薄 |
| **失忆症** | 83/85 tool_result 被清理 | 早期工作白做，重复犯错 |
| **膨胀-重置循环** | 8 次 compact | 工作中断，连续性差 |
| **决策疲劳** | 后期 +70K 膨胀 | 后期决策质量下降 |

**最务实的改进路径**：

1. **短期**：在 system prompt 中强制「思考-行动分离」和「批量处理」
2. **中期**：实现里程碑自动 compact + 状态外置到工作区文件
3. **长期**：提供批量工具接口（ReadDir、BatchExec），从根本上减少往返次数

当前 85 轮纯工具循环的模式，本质上是一个 **「没有短期记忆、只能机械执行」的 Agent**。如果不改变这个行为模式，无论后端多强、上下文窗口多大，最终都会回到同样的膨胀-崩溃循环。
