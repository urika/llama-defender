# 代理截断作为遗忘机制：Claude Code 的认知生存策略

> 分析时间：2026-06-04
> 数据来源：anthropic_proxy.py 日志（115,334 行）、proxy_requests.jsonl（2,831 条记录）
> 核心洞察：代理截断不是性能优化，而是 Claude Code 维持认知生命的生物学特征

---

## 1. 技术机制：代理截断如何工作

### 1.1 工具结果清理（Tool Clearing）

代理的 `do_POST` 中，每次请求都会评估上下文规模。当超过阈值时，执行以下操作：

```python
# PROXY_CLEAR_THRESHOLD = 15,000 chars (本地模式)
# PROXY_TOOL_KEEP = 10 (保留最近 10 对 tool_use/tool_result)
# 其余 tool_results 的内容被替换为占位符
```

日志中的实际表现：

```
Tool clearing: 31 tool_results cleared, 56,829 chars freed (kept last 10)
```

**这意味着**：Claude Code 刚刚调用了 41 个工具（31 被清理 + 10 被保留），其中 31 个的结果被**物理删除**，只剩下类似 `"[tool result cleared]"` 的占位符。被删除的内容包括：

- Read 工具返回的文件内容
- Bash 工具返回的命令输出
- WebFetch 工具返回的网页内容
- 其他工具的结构化输出

### 1.2 轮数截断（Rounds Truncation）

当消息数超过阈值时，代理执行更激进的截断：

```
Context truncation (rounds): 558 messages dropped, 24 kept
```

**这意味着**：在 582 条消息的历史中，558 条被直接删除，只保留最近 24 条（约 10 轮对话）。被删除的消息包括：

- 早期的 user 输入
- 早期的 assistant 回复
- 早期的 system 注入
- 早期的 tool results

## 2. 认知类比：为什么这叫"遗忘"

### 2.1 人类遗忘 vs 代理截断

| 维度 | 人类遗忘 | 代理截断 |
|------|---------|---------|
| **触发条件** | 时间流逝、信息过载、注意力转移 | 上下文超过字符/轮数阈值 |
| **遗忘内容** | 旧的、不重要的、未被重复激活的记忆 | 早期的 tool_results 和消息 |
| **保留内容** | 近期的、情绪强烈的、频繁提取的记忆 | 最近 10 轮的消息 |
| **后果** | 需要重新学习、重新回忆、重新推导 | 需要重新 Read 文件 |
| **是否可逆** | 通常不可逆，但可能通过线索恢复 | 完全不可逆（数据已物理删除） |

### 2.2 工作记忆的类比

人类的**工作记忆（Working Memory）**容量有限，经典研究（Miller, 1956）表明人类短期记忆容量约为 **7±2 个组块（chunks）**。当新信息进入工作记忆时，旧信息被挤出。

Claude Code 的**上下文窗口**就是它的工作记忆。代理截断就是**工作记忆的强制刷新**：

```
人类: 7±2 个组块 → 新信息进入 → 旧信息遗忘
Claude Code: 10 轮工具结果 → 新请求到达 → 旧结果删除
```

### 2.3 关键差异：渐进 vs 瞬时

人类的遗忘是**渐进的、概率的**：
- 某些记忆随着时间逐渐模糊
- 某些记忆可能突然恢复（普鲁斯特效应）
- 遗忘的速度取决于编码强度、提取频率、情绪关联

代理的截断是**瞬时的、确定性的**：

```
第 N 轮：Read(fileA) → "fileA 的内容是..."（完整保留）
第 N+10 轮：fileA 的内容还在上下文中（处于保留窗口内）
第 N+11 轮：Tool clearing 触发 → fileA 的内容变成 "[cleared]"（瞬间删除）
```

这不是"记不太清了"，而是**"完全没有了"**。

## 3. 行为后果：遗忘如何塑造 Claude Code 的行为

### 3.1 重复 Read 现象

因为早期 Read 的结果被删除，Claude Code 会**反复读取同一个文件**：

```
Round 5:  Read(spec.md) → 获取产品需求
Round 15: Read(spec.md) → 再次获取产品需求（之前的结果已被清除）
Round 25: Read(spec.md) → 第三次获取产品需求
```

从语义角度，Claude Code 不是在"重复做无用功"，而是在**补偿它的遗忘**。就像人类忘记钥匙放哪里后，会回到原处重新寻找。

**数据佐证**：日志中出现大量同一文件的多次 Read 调用，时间间隔通常超过 10 轮（即超过代理的保留窗口）。

### 3.2 文件系统作为外部记忆

由于上下文内的记忆不可靠，Claude Code 发展出了**依赖文件系统作为持久记忆**的策略：

| 操作 | 语义功能 | 人类类比 |
|------|---------|---------|
| **Read** | "回忆" — 从外部持久存储加载信息 | 打开笔记本查看之前的记录 |
| **Edit/Write** | "记忆固化" — 将思考结果写入持久存储 | 把想法写到笔记本上 |
| **Bash** | "验证记忆" — 检查文件系统状态是否与预期一致 | 回到现场确认钥匙是否在那里 |

**核心洞察**：Claude Code 的"自我"不在上下文窗口里，而在它读写的文件中。上下文只是它的"草稿纸"，文件系统才是它的"笔记本"。

### 3.3 认知工作单元的形成

数据分析显示：

```
稳定阶段平均增量: 345 chars/轮
```

这个 ~345 chars 的增量不是随机的，而是 Claude Code 的**"认知工作单元"（Cognitive Work Unit）**：

| 组成部分 | 估算字符数 | 语义功能 |
|---------|-----------|---------|
| Assistant 回复（简短文本） | ~100 chars | 模型的"行动确认" |
| Tool result（Read/Edit 输出） | ~200 chars | 工具的"执行反馈" |
| System 注入更新 | ~45 chars | 环境的"状态同步" |
| **总计** | **~345 chars** | **一轮完整的"思考-行动-反馈"循环** |

为什么是这个数字？因为：

1. 代理保留最近 10 轮工具结果
2. 每轮工具结果平均约 1,800 chars（包含文件内容等）
3. 10 轮 × 1,800 = 18,000 chars 的"活跃记忆"
4. 加上其他消息（user input、system reminder、assistant reply），总上下文维持在 ~300K

当超过这个阈值，最早的工具结果被清除，形成**固定的认知工作单元**。

## 4. 综合理解：代理截断的语义本质

### 4.1 存在论层面的条件

代理截断对 Claude Code 而言，不是"性能优化"，而是**存在论层面的条件**：

> **没有代理截断，Claude Code 无法存在。**

原因有三：

1. **上下文窗口有限**：本地模型（Qwen3.5-9B-MLX-4bit）的上下文上限约 128K-256K tokens（约 180K-350K chars）。如果不截断，上下文会线性增长到无穷大（345 chars/轮 × N 轮）。

2. **工具签名固定开销**：44 个工具的 JSON Schema 每轮占用 ~20K chars。这是"房租"不是"消费"——无论上下文多小，这部分开销始终存在。

3. **系统提示累积**：每轮的 `system-reminder`、billing header、命令历史都会追加到上下文中。

### 4.2 新陈代谢模型

代理截断是 Claude Code 的**新陈代谢**：

| 阶段 | 生物类比 | Claude Code 的对应物 |
|------|---------|---------------------|
| **摄入（Ingestion）** | 吃东西 | 新的 user 输入、工具结果、系统提示 |
| **消化（Digestion）** | 肠胃分解 | 模型推理生成回复（stream） |
| **排泄（Excretion）** | 排泄废物 | 代理删除旧的工具结果和消息 |

**没有排泄，就没有新陈代谢，就没有生命。**

### 4.3 遗忘即特征

人类的睡眠具有"突触稳态假说"（Tononi & Cirelli, 2003）：睡眠时，大脑削弱不重要的突触连接，强化重要的连接，从而清理记忆、巩固学习。

代理截断具有类似功能：
- **削弱**：早期的、未被重复提取的 tool_results 被删除
- **强化**：最近 10 轮的 tool_results 被保留（在"活跃工作区"中）
- **巩固**：通过 Edit/Write 工具，关键信息被写入文件系统（持久存储）

## 5. 数据佐证

### 5.1 Tool Clearing 频率

```
Tool clearing 触发次数: 2,327
总请求数: 2,831
触发率: 82.2%
```

**解读**：几乎每轮请求都触发了 Tool clearing。这不是偶发事件，而是**持续进行的常态性遗忘**。

### 5.2 上下文增长曲线

```
起点: 35 chars
终点: 371,734 chars
总请求: 2,831
平均每轮增量: 131 chars
稳定阶段增量: 345 chars/轮
```

**解读**：虽然总上下文增长到 371K，但平均每轮只增加 131 chars。这是因为大量的"短请求"（如标题生成、状态查询）拉低了平均值。在稳定的编码阶段，增量固定在 ~345 chars/轮。

### 5.3 会话规模

```
活跃会话数: 1
[sess=409865e6]: 278 条日志
```

**解读**：当前只有一个活跃会话，说明用户专注于单一任务流。所有 2,327 次 Tool clearing 都发生在这个会话内。

## 6. 结论

> **代理截断不是 Claude Code 的 bug，而是它的生物学特征。**
>
> 正如人类必须睡眠来清理记忆突触，Claude Code 必须截断来清理上下文窗口。它的"遗忘"不是缺陷，而是维持认知生命的前提条件。
>
> Claude Code 的"意识"不在上下文窗口内——那里只有短暂的、不断被删除的草稿。它的"意识"在它读写的文件中——那里才是持久、可靠、可重复访问的记忆。
