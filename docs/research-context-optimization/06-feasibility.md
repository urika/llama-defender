# 06 可行性分析：当前机器与模型能否支撑编程 Agent

## 6.1 场景目标定义

用户期望的终极场景：

> 在 **MacBook Pro M5 Pro / 48 GB unified memory** 上，通过本地部署 **Qwen3.6-35B-A3B via Rapid-MLX**，运行 Claude Code 风格的编程 Agent，完成代码编写、修改、调试、重构等任务。

关键质量要求：
- 单次 Agent 会话可持续 **30 分钟以上** 不出现死循环或 OOM
- 能够处理 **中小型代码库**（约 1–5 万行有效代码）
- 支持多文件 Read/Write/Edit/Bash 工具链
- 响应延迟可接受（首 token < 10s，生成速度 > 10 tok/s）

## 6.2 当前约束条件

| 维度 | 实际情况 |
|------|----------|
| 硬件 | Apple M5 Pro，48 GB unified memory |
| 模型 | Qwen3.6-35B-A3B，4-bit 量化 via Rapid-MLX |
| 权重占用 | 约 18–20 GB（35B × 4-bit ≈ 17.5 GB + 开销） |
| 可用余量 | 操作系统、IDE、浏览器等约占用 8–12 GB，留给模型/KV cache 约 16–20 GB |
| 上下文窗口 | 配置 131,072 tokens，但实际可用受 KV cache 内存限制 |
| 后端 | rapid-mlx v0.6.30，存在忽略 `max_tokens`、TurboQuant cache persist 失败等问题 |
| 客户端 | Claude Code，携带 40+ 工具定义，会话历史快速增长 |

## 6.3 可行性判断

### 6.3.1 总体结论

**可行，但有显著边界。当前机器和模型组合可以支撑轻度到中度编程 Agent 场景，但无法无限制地支撑大型代码库或长时间复杂会话。**

更精确地说：

| 场景 | 当前可行性 | 优化后可行性 |
|------|------------|--------------|
| 单文件修改 / 脚本编写 | ✅ 可行 | ✅ 稳定可行 |
| 3–5 个文件的小功能开发 | ⚠️ 基本可行 | ✅ 可行 |
| 中小型项目（1–5 万行）日常维护 | ⚠️ 有挑战 | ⚠️–✅ 基本可行 |
| 大型 monorepo（10 万行+）导航与重构 | ❌ 不可行 | ❌ 仍不可行 |
| 2 小时以上的连续复杂会话 | ❌ 不可行 | ⚠️ 可能延长到 1 小时左右 |
| 需要保持完整上下文的跨文件重构 | ❌ 不可行 | ❌ 仍不可行 |

### 6.3.2 为什么大型/长时会话不可行

1. **KV cache 内存硬约束**
   - 35B 模型 + 131K 上下文，KV cache 在 FP16 下可达 **10–15 GB**。
   - 加上权重 18–20 GB，仅模型部分就可能接近 30 GB。
   - 48 GB 统一内存中其他进程（Claude Code、IDE、浏览器）占用后，留给模型的余量有限，OOM 风险持续存在。

2. **上下文增长不可逆**
   - Agent 会话中每次工具调用都会增加 messages。
   - 即使优化后压缩 30–40%，长会话仍会逐步逼近窗口上限。
   - 一旦触发激进截断，语义丢失 → 循环风险上升。

3. **模型能力边界**
   - 35B 4-bit 量化模型在复杂推理、长距离依赖、自我纠正方面弱于 Claude 3.5 Sonnet / GPT-4。
   - 本地模型更容易陷入重复动作，需要代理层大量代偿逻辑。

4. **后端成熟度**
   - rapid-mlx 作为较新的后端，稳定性和功能完整性不如 llama.cpp 成熟分支。
   - `max_tokens` 被忽略、cache persist 失败等问题会直接影响 Agent 体验。

## 6.4 优化后的预期效果

如果完整落地 `04-solutions.md` 中的方案，预期可以：

| 指标 | 当前状态 | 优化后预期 |
|------|----------|------------|
| prefix cache 命中率 | ~0% | 30–50%（局部） |
| 单轮 token 量 | 基准 | 下降 20–35% |
| wasted/re-read 死循环 | 频繁（曾达 219 次） | 显著减少，偶发 |
| 会话稳定时长 | 10–30 分钟 | 30–60 分钟 |
| OOM 频率 | 较高 | 降低但未消除 |
| 可用代码库规模 | <1 万行 | 1–5 万行 |

**关键认知**：优化能显著改善体验、延缓瓶颈出现，但**不能突破 48GB 内存和 35B 模型能力的物理/算法天花板**。

## 6.5 若目标不可行，需要修改哪些条件

如果用户期望的是"无限制地像 Claude Code + Claude API 一样使用本地模型"，则当前条件不可行，需要修改以下一个或多个条件：

### 条件 1：硬件升级（最有效）

| 升级方向 | 效果 | 成本 |
|----------|------|------|
| 64 GB unified memory | 可支撑更长上下文或更稳定并发 | 中 |
| 128 GB unified memory | 基本消除内存焦虑，可尝试更大模型或更长上下文 | 高 |
| 外接 GPU / 多台 Mac | 分摊负载，但当前架构不支持分布式 | 高且复杂 |

**结论**：升级到 64GB 可明显改善；128GB 才能接近"无限制本地 Agent"的体验。

### 条件 2：换用更小的模型

| 模型选项 | 优势 | 劣势 |
|----------|------|------|
| Qwen3.6-9B | 内存占用小，速度快，上下文可更长 | 代码能力明显弱于 35B |
| Qwen3.6-14B/27B | 平衡性能与资源 | 仍受 48GB 约束，但压力更小 |
| 专用代码小模型（如 DeepSeek-Coder-V2-Lite） | 代码场景优化 | 通用 Agent 能力可能不足 |

**结论**：若优先考虑稳定性和长会话，可降级到 14B/27B；若优先考虑代码质量，保留 35B 但接受会话长度限制。

### 条件 3：架构变化

| 方案 | 说明 | 复杂度 |
|------|------|--------|
| 混合云-本地模式 | 复杂任务上云，简单/敏感任务本地 | 中（项目已支持 cloud 模式） |
| 分片 Agent | 不同子任务用不同模型/实例 | 高 |
| 长期记忆层 | 跨会话持久化文件索引、项目知识 | 高 |
| 代码检索增强（RAG） | 不把所有代码塞进上下文，按需检索 | 中 |

**结论**：混合模式是当前最现实的扩展路径；RAG 是处理大型代码库的有效补充。

### 条件 4：降低使用强度

| 调整 | 效果 |
|------|------|
| 每 30 分钟主动重启一次 Agent 会话 | 清空累积上下文，释放 KV cache |
| 限制每次任务范围（单次 1–3 个文件） | 降低上下文增长速度 |
| 人工介入切分复杂任务 | 避免模型一次性处理过长推理链 |

## 6.7 推荐的使用模式

基于以上分析，建议将当前机器+模型定位为：

> **"本地轻量编程 Agent 平台"，适合中小任务、隐私敏感场景、云 API 不可用时的 fallback。**

具体使用建议：

1. **任务切分**：将大任务拆分为 15–30 分钟可完成的子任务。
2. **定期重启**：每完成一个子任务或 30 分钟后重启会话，释放 KV cache。
3. **文件聚焦**：每次交互前用 `Glob`/`Grep` 定位相关文件，避免模型在无关文件中漫游。
4. **云地混合**：复杂架构设计、大型重构、跨文件推理使用 cloud 模式；代码实现、敏感文件修改使用 local 模式。
5. **启用全部优化**：Cache Aligner、语义压缩、循环防御、资源护栏应作为 local 模式默认配置。
6. **设置预期**：不追求与 Claude API 同等的无限上下文能力，而是发挥本地部署的低延迟、隐私、无计费优势。

## 6.8 最终结论

| 问题 | 答案 |
|------|------|
| 当前机器+模型能否支撑编程 Agent？ | **能，但有明显边界。** |
| 优化后能否达成目标？ | **能达成"轻量到中度"编程 Agent 目标，但无法达成"无限制大型项目 Agent"目标。** |
| 不可行时应修改的条件 | 硬件升级（64GB/128GB）、换更小模型、引入 RAG/混合云架构、降低使用强度 |
| 当前最适合的场景 | 单文件/小项目开发、代码 Q&A、脚本与数据处理、配置文件管理、命令行辅助 |
