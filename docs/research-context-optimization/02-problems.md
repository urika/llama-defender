## 2.2 八大核心痛点

### 痛点 1：上下文长度控制 vs prefix cache 命中率的根本性冲突

**现象**
- 原始报文相邻请求共同前缀 **97%**。
- 代理截断后，共同前缀跌至 **19%–35%**。
- Rapid-MLX prefix cache 命中率跌至 **0%**。

**根因**
1. `rounds` 策略轮次边界不固定，新轮次加入时旧轮次被挤出，导致消息序列重组而非滑动。
2. Tool Clearing 将旧 `tool_result` 替换为占位符，改变 token 序列。
3. 动态 system 消息（如 `The date has changed...`、`task tools reminder`）插入对话中间，导致索引偏移。
4. Token budget 动态削减，字符数超限时 `keep_rounds` 从 8 减到 7，额外挤出一整轮。

**业务影响**
- 本地后端重复计算成本高：无 cache 命中意味着每次请求都要完整 prefill。
- 长会话体验差：上下文越大，prefill 越慢，形成"越聊越卡"的恶性循环。
- 配置调参困难：增大窗口 → OOM 风险；减小窗口 → cache 命中率更低。

---

### 痛点 2：Tool Result 清除策略的语义损失与 re-read 死循环

**现象**
- 关闭 Tool Clearing 前：`wasted` 错误从 7→9→11→13 持续增长，最终出现 219 次死循环。
- 模型无法区分 `"[cleared: 21000 chars]"` 与 `"读取失败"`。
- 后端返回 `"Wasted call"` 后，模型继续 `Read` 同一文件。

**根因**
1. 清除是删除而非压缩：旧实现将 `tool_result` 替换为通用占位符，模型丢失文件内容记忆。
2. 错误信息语义不被模型理解：`Wasted call` 是 Claude Code 缓存机制提示，Qwen 模型无法将其理解为"不要重新读取"。
3. 无循环检测：旧代理无法在重复行为发生时主动干预。

**业务影响**
- 任务完全停滞：死循环期间模型产出为零，用户只能手动重启。
- 上下文恶性膨胀：每次 `Read + Wasted call` 增加一对消息，加速窗口耗尽。
- 用户信任下降：长会话频繁进入不可恢复状态。

---

### 痛点 3：循环行为的多样性与防御军备竞赛

**现象**
代理层需应对多种循环类型：

| 循环类型 | 触发工具 | 根因 |
|---|---|---|
| 错误驱动循环 | Read | 不理解 `Wasted call` |
| 认知驱动循环 | Write | 模型内部认知矛盾（"需要精简" vs "已精简"） |
| 文本输出循环 | 无（纯文本） | 重复输出相同段落 |
| Bash 去重循环 | Bash | 已清空内容触发 Jaccard 相似度匹配 |
| Blocker 循环 | 任意 | 连续相同错误类型 |

**根因**
- 模型自我纠正能力极弱：Read 死循环需要 219 次失败才学会用 `Bash cat`。
- 客户端架构复杂：Claude Code 主进程/只读子代理/执行子代理工具集不同。
- 跨请求状态丢失：每次请求从 Level 0 开始，循环历史不被继承。

**业务影响**
- 代理层不断追加补丁：从 L1/L2 到 L3，从工具循环到文本循环。
- 配置项激增：`PROXY_LOOP_THRESHOLD`、`PROXY_LOOP_LEVEL2`、`PROXY_LOOP_LEVEL3`、`PROXY_TEXT_LOOP_*` 等。
- 误伤风险：过度干预可能打断正常多步操作。

---

### 痛点 4：后端资源约束的持续性压力

**现象**
- `Metal memory limit (499000) exceeded` OOM 错误。
- rapid-mlx 运行 7 分钟后 tok/s 从 56 跌至 12（衰减 78%）。
- `--gpu-memory-utilization > 0.85` 触发 Apple Silicon kernel panic 警告。
- TurboQuant 虽降低内存，但导致 cache persist 失败。

**根因**
- 48GB unified memory 不是无限的：35B 模型 + KV cache + prefix cache + 其他进程共享内存。
- 代理无法精确预估后端实际内存使用：`chars/4` 是粗略估算。
- 后端行为不透明：rapid-mlx v0.6.30 忽略 `max_tokens`，TurboQuant 不支持 `state` 属性。

**业务影响**
- 服务中断：OOM 导致后端崩溃，代理返回 503/504。
- 性能不可预测：用户感受到"越用越慢"。
- 配置保守化：被迫降低并发、限制窗口，牺牲性能换稳定。

---

### 痛点 5：工具过滤白名单困境

**现象**
- `TOOL_ALWAYS_KEEP` 白名单持续扩展，每次 Claude Code 升级都可能漏掉新工具。
- 工具过滤后 prefix cache 断裂：不同请求保留的工具列表顺序不同。
- `recent=0` 的统计一度被误认为 bug，实则是白名单已覆盖大部分常用工具。

**根因**
- 白名单是被动防御：Claude Code 新工具层出不穷，无法提前枚举。
- 过滤结果不稳定：按输入顺序保留导致不同请求工具集顺序不同。
- 自动晋升机制被移除：`_tool_freq` 全局计数器因实现缺陷被删除。

**业务影响**
- 新工具首次使用失败：用户报告 → 紧急加白名单 → 再升级再失败。
- cache 命中率进一步下降：工具顺序变化破坏前缀稳定。
- 可观测性需求增加：必须记录 `filtered_out` 列表才能诊断。

---

### 痛点 6：客户端与后端兼容性摩擦

**现象**
- Claude Code `mid-conversation-system` beta 注入 system 消息，Qwen chat template 报错 `System message must be at the beginning`。
- rapid-mlx 返回空 SSE 流，代理超时。
- Anthropic SDK 的某些消息格式（如 `developer` role）不被后端识别。
- 后端 rapid-mlx v0.6.30 忽略 `max_tokens`，导致输出截断、JSON 损坏。

**根因**
- 客户端超前于后端：Claude Code 支持 Anthropic 最新 API 特性，本地后端跟不上。
- 模型模板不标准：Qwen 官方 chat template 对 system 消息位置要求严格。
- 代理承担协议翻译风险：Anthropic → OpenAI 格式转换中可能丢失或误传语义。

**业务影响**
- 启动失败/请求超时：用户以为是代理问题，实际是后端模板不兼容。
- 需要人工修复模板：每次新模型/新后端版本都要检查 chat template。
- 输出质量受损：`max_tokens` 被忽略导致 force_stop，JSON tool_call 参数截断。

---

### 痛点 7：可观测性不足

**现象**
- `re_read_rate` 公式错误导致 2862% 的异常值。
- `/status` 轮询产生大量日志噪音。
- 工具过滤 `recent=0` 被误判为 bug。
- 500 错误率 22% 但无法快速定位根因。

**根因**
- 指标定义不清晰：`re_read_rate` 分子用了 `total_reads` 而非 `re_read_files`。
- 日志粒度不一致：代理日志、后端日志、metrics JSONL 格式不统一。
- 缺少实时看板：`/status` 只有当前状态，无历史趋势。
- 调试信息不足：大请求失败时缺少原始请求快照。

**业务影响**
- 问题发现滞后：依赖用户报告或事后日志分析。
- 修复验证困难：无法快速确认修复是否生效。
- 调参盲目：不知道哪个参数对指标影响最大。

---

### 痛点 8：配置复杂度与运维负担

**现象**
- `anthropic_proxy.py` 已有 **31+ 个 env vars**。
- `configs/*.conf` 中代理相关参数与后端参数混合。
- 同一功能有多种策略（`char`/`rounds`/`fifo`/`smart`），选择困难。
- 参数之间存在隐性耦合（如 `PROXY_CLEAR_ENABLED` 与 `PROXY_CTX_TRUNCATE_STRATEGY`）。

**根因**
- 功能快速迭代：每个痛点都通过新增参数解决。
- 缺少配置分层：全局默认值、后端默认值、场景覆盖值混杂。
- 文档跟不上代码：`AGENTS.md` 警告 25+ 参数缺少推荐组合。

**业务影响**
- 新用户上手困难：不知道该如何配置。
- 调参成本高：需要理解每个参数的含义和相互作用。
- 回归风险：修改一个参数可能意外影响其他功能。

## 2.4 本地部署与机器性能对痛点的约束影响

当前项目运行在 **MacBook Pro M5 Pro / 48 GB unified memory**，使用 **Qwen3.6-35B-A3B via Rapid-MLX**（偶尔 Qwen3.6-27B-MTP）。这一硬件与模型组合对痛点的产生、放大与治理空间有决定性影响。

### 2.4.1 约束来源

| 约束维度 | 具体情况 |
|----------|----------|
| 内存上限 | 48 GB 统一内存被操作系统、IDE、浏览器、模型、KV cache、prefix cache 共享 |
| 模型规模 | 35B 4-bit 量化模型本身占用约 18–20 GB，留给上下文与 cache 的余量有限 |
| 后端能力 | rapid-mlx v0.6.30 存在忽略 `max_tokens`、TurboQuant 不支持 cache persist、模板兼容性弱等问题 |
| 模型能力 | 本地 Qwen 模型在指令跟随、错误提示理解、自我纠正方面弱于 Claude API |
| 成本结构 | 本地运行无按 token 计费，但计算与内存成为稀缺资源 |

### 2.4.2 痛点与约束的关联分类

| 分类 | 包含痛点 | 说明 |
|------|----------|------|
| **直接由约束导致** | 痛点 1、4、5 | 内存不足 → 必须截断/过滤 → 引发前缀破坏与 OOM |
| **被约束显著放大** | 痛点 2、3、6 | 本地模型理解力弱、后端行为不一致，使语义丢失更快演变成死循环 |
| **相对独立但治理受约束影响** | 痛点 7、8 | 可观测性与配置问题本身与硬件无关，但正因为资源紧张才需要大量精细参数 |

### 2.4.3 各类痛点在约束下的具体表现

#### 1) 直接由约束导致

**痛点 1：上下文 vs prefix cache 冲突**
- 若硬件内存无限，可直接保留完整历史，无需激进截断，prefix cache 命中率自然高。
- 现实是 48GB 内存下，35B 模型 + KV cache 已占用大部分空间，代理必须主动截断以控制单轮 token 量。
- 截断策略（`rounds`、`fifo`、`char`）改变了相邻请求的 token 序列，导致 Rapid-MLX prefix cache 命中率从理论值 ~97% 跌至实测 **0%**。

**痛点 4：后端资源约束**
- 这是约束本身：`Metal memory limit (499000) exceeded`、运行 7 分钟后 tok/s 衰减 78%、TurboQuant cache persist 失败等，都是 35B 模型在 48GB 机器上的直接后果。
- 代理层所有上下文优化都必须在此硬边界内完成。

**痛点 5：工具过滤白名单困境**
- Claude Code 每次请求携带 40+ 工具定义，schema 本身可能占数千 token。
- 若上下文无限，无需过滤工具；正因为本地窗口紧张，才被迫压缩工具列表。
- 过滤又破坏了固定前缀，进一步降低 cache 命中率。

#### 2) 被约束显著放大

**痛点 2：tool result 清除与 re-read 死循环**
- 表面是"清除策略错误"，深层是本地模型无法像 Claude API 那样理解 `Wasted call` 这类客户端缓存提示。
- 实测中 Qwen 模型需要 **219 次重复** 才学会换用 `Bash cat`，而在 Claude API 上几乎不会发生。
- 内存压力又迫使代理必须清除旧 tool result，加速了死循环的形成。

**痛点 3：循环行为多样性**
- 模型能力越弱，代理层需要越多防御机制（L1/L2/L3、blocker、text loop、Jaccard 检测）。
- 这些补丁在云 API 上很多是不必要的，但在本地部署中是保命手段。

**痛点 6：客户端/后端兼容性摩擦**
- Claude Code 快速迭代 Anthropic API 新特性，而 rapid-mlx 与 Qwen chat template 更新滞后。
- 若使用 Claude 官方 API，代理无需承担大量协议翻译与模板修正工作。
- 本地部署使代理层成为"兼容性缓冲带"。

#### 3) 相对独立但治理受约束影响

**痛点 7：可观测性不足**
- 指标设计错误（如 `re_read_rate`）与部署方式无关。
- 但本地环境复现长会话成本高，使错误指标更难被发现和修正。

**痛点 8：配置复杂度**
- 31+ 个环境变量的存在，很大程度上是因为要在有限资源下做精细权衡（context limit、tool keep、cache、concurrency、compression 等）。
- 若换成云 API 1M token 上下文，约一半的参数可以收敛或消失。

### 2.4.4 对照验证：云模式下的痛点变化

项目支持 `./manage.sh start-cloud` 切换到 DeepSeek/OpenAI API，这提供了一个天然对照组：

| 痛点 | 云模式下是否减弱 | 原因 |
|------|------------------|------|
| 痛点 1 | 显著减弱 | 1M token 上下文允许保留更完整历史，截断压力小 |
| 痛点 2 | 显著减弱 | 模型理解力强，极少出现 219 次死循环 |
| 痛点 3 | 显著减弱 | 自我纠正能力强，代理防御机制需求降低 |
| 痛点 4 | 基本消失 | 计算在云端，本地无 OOM 与性能衰减 |
| 痛点 5 | 减弱 | 上下文充裕，无需激进工具过滤 |
| 痛点 6 | 部分减弱 | 后端原生支持 Anthropic/OpenAI 协议，兼容性摩擦减少 |
| 痛点 7 | 不变 | 指标设计问题与部署方式无关 |
| 痛点 8 | 部分减弱 | 大量资源相关参数可收敛，但配置治理仍是工程问题 |

### 2.4.5 对后续优化的启示

1. **区分"资源约束导致的症状"与"代理层应承担的根因"**
   - 资源约束是背景，不能消除；但代理层可以通过 Cache Aligner、语义保留压缩等手段在约束内做得更好。
   - 不能把所有问题都归咎于硬件，否则会导致过度工程化。

2. **优先做不依赖硬件升级的优化**
   - Cache Aligner（痛点 1）、语义保留压缩（痛点 2）、错误指标修复（痛点 7）与硬件无关，应优先落地。

3. **明确功能的环境假设**
   - 某些策略（如 aggressive tool filtering）只在 local 模式有意义，cloud 模式应默认关闭。
   - 配置模板应区分 `local-balanced`、`local-aggressive`、`cloud`。

4. **保留云模式作为效果对照**
   - 用 `./manage.sh start-cloud` 验证同一痛点在云端的严重程度，避免为"硬件问题"过度设计代理层方案。

5. **承认硬件升级的边界收益**
   - 即使升级到 128GB 内存并使用更小模型，痛点 6（兼容性）、7（可观测性）、8（配置复杂度）仍需工程化解决。
   - 但痛点 1–5 的大部分症状会显著缓解。
