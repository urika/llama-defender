# 01 需求：代理层上下文优化

## 1.1 项目背景

当前项目是一个 Anthropic-compatible API 代理层，核心数据流为：

```
Claude Code → anthropic_proxy.py:4000 → rapid-mlx/llama-server:8081 → GGUF/MLX 模型
```

运行环境：
- 设备：MacBook Pro M5 Pro，48 GB unified memory
- 主力模型：Qwen3.6-35B-A3B via Rapid-MLX（偶尔 Qwen3.6-27B-MTP via llama-server）
- 使用场景：长时 agentic coding 会话
- 客户端：Claude Code

本地运行无按 token 计费，但受限于 **内存、上下文窗口、推理延迟**。因此，"成本"的核心是上下文压力与计算资源消耗。

## 1.2 北极星目标

> **在保持或提升 Agent 任务完成率的前提下，降低长会话的上下文压力，延缓上下文窗口耗尽，减少模型因噪声产生的无效动作。**

## 1.3 二级目标

| 目标编号 | 目标描述 | 衡量方式 | 合格线 | 优秀线 |
|----------|----------|----------|--------|--------|
| G1 | 降低单轮请求 token 量 | 每轮请求 `input_chars` / 估算 tokens | 下降 ≥ 15% | 下降 ≥ 35% |
| G2 | 提高 prefix cache 命中率 | 相同 system prompt 连续请求的 latency / cache hit 比例 | 有提升 | 明显提升 |
| G3 | 减少长会话中的无效循环 | `wasted`、`loop`、`blocker` 类错误计数 | 不增加 | 减少 |
| G4 | 保持或提升任务完成率 | 端到端 benchmark / 人工抽检 | 下降 ≤ 5% | 持平或提升 |
| G5 | 保持可观测与可调试 | metrics 日志完整性、错误可追踪 | 基础覆盖 | 完整覆盖 |
| G6 | 控制实现与维护成本 | 新增代码量、依赖数、测试覆盖 | ≤ 3 天/项 | ≤ 1 天/项 |

## 1.4 非目标

- **不追求最大压缩率**：外部产品对 AWS CLI 可达 66%，但本场景以代码文件、对话历史、tool result 为主，不能简单套用。
- **不改变客户端交互模型**：Claude Code 仍通过 `127.0.0.1:4000` 连接，不引入额外配置步骤。
- **不做跨会话持久记忆**：本次聚焦单次会话内的上下文优化，不涉及长期记忆层。
- **不替换当前代理架构**：不引入 Kompact/TokenSieve 独立服务，而是借鉴其具体实现嵌入当前代理。

## 1.5 关键约束

| 约束 | 说明 | 影响 |
|------|------|------|
| 标准库优先 | `anthropic_proxy.py` 当前无第三方依赖 | 引入新功能应尽量保持低依赖 |
| 本地后端脆弱 | rapid-mlx 存在 OOM、性能衰减、模板兼容性等问题 | 代理层改动不能加剧不稳定 |
| 单人维护 | 项目由少数人维护 | 方案需简单、可回滚、易调试 |
| 双模式运行 | 需同时支持 local 和 cloud 模式 | 压缩策略应可区分模式启用 |
| 生产验证难 | 真实长会话成本高 | 需要单元/集成测试覆盖核心路径 |

## 1.6 成功标准总结

1. 长会话（30 分钟以上）的上下文增长可控，不出现 OOM 或严重性能衰减。
2. 相同 system prompt 的连续请求 prefix cache 命中率提升。
3. `wasted call`、`re-read` 等无效循环指标不增加或下降。
4. 任务完成率不下降，关键信息（代码文件、错误信息）不丢失。
5. 每次新功能都有独立的开关、metrics 记录和回归测试。
