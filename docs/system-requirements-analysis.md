## 二、问题与需求提炼

### 需求领域 1: 上下文容量不足

**现象**:
- 48GB 统一内存 + 35B 模型，实际可用于 KV cache 的上下文有限
- Agentic coding 10 轮 = 90+ 消息，轻松超出 50K token 实际容量
- Claude Code 每次请求发送全量对话历史，不区分 token 预算

**根因**: 本地模型上下文窗口是硬约束（48GB 物理限制），与 Claude Code 的"全量历史"模式不兼容。

**需求点**:

| ID | 需求 | 描述 | 优先级 |
|----|------|------|--------|
| R1.1 | **主动上下文截断** | 在 prompt 超过 token 预算时，主动丢弃旧消息而非让模型崩溃 | P0 |
| R1.2 | **压缩摘要替代** | 被丢弃的消息不直接消失，而是生成结构化摘要保留关键信息 | P0 |
| R1.3 | **增量压缩** | 避免每次全量重压缩，只处理新增被截断消息 | P1 |
| R1.4 | **关键词按需检索** | 保留被截断消息的关键词索引，在当前上下文相关时注入 | P1 |

### 需求领域 2: 循环与失控

**现象** (Session d278d9eb 实测):
- 模型进入 Read-Clear-ReRead 死循环，持续 5 小时
- `max_run=25`，循环检测注入 break message 但模型完全忽略
- 工具清除 Read 内容 → 模型认为"丢失" → 重新读取 → 再清除 → 无限循环
- 每轮 ~20s TTFT，chars 从 154K 增长到 328K

**根因**: 本地 LLM 的 instruction following 能力弱于 Claude/GPT-4，无法可靠响应"软性"干预。同时工具清除与模型行为形成了负反馈循环。

**需求点**:

| ID | 需求 | 描述 | 优先级 |
|----|------|------|--------|
| R2.1 | **递进式循环干预** | 从软性提示到硬性工具移除到强制纯文本，三级递进 | P0 |
| R2.2 | **智能清除策略** | 清除 Read 内容时保留预览片段，降低模型重读欲望 | P0 |
| R2.3 | **近期 Read 保护** | 最近被 Read 的文件内容获得额外评分，不易被清除 | P0 |
| R2.4 | **阻塞模式检测** | 检测连续相同错误类型（如连续 FileNotFoundError）并干预 | P1 |

### 需求领域 3: 延迟优化

**现象**:
- 无缓存: 68K tokens → 90.8s TTFT
- Prompt 构成: 44 tools (~10K) + 81 messages (~58K)
- 每次请求重新计算所有 token，无跨请求复用

**根因**: 模型推理延迟与 prompt 长度线性相关。固定开销（tools, system prompt）在每个请求中重复计算。

**需求点**:

| ID | 需求 | 描述 | 优先级 |
|----|------|------|--------|
| R3.1 | **KV Cache 复用** | 利用推理引擎的 prefix cache 能力，避免重复计算稳定前缀 | P0 |
| R3.2 | **前缀稳定化** | 日期标准化、固定占位文本，确保前缀跨请求不变 | P0 |
| R3.3 | **工具定义过滤** | 从 44 个工具中筛选实际需要的子集，减少 ~5-8K tokens | P0 |

### 需求领域 4: 模型兼容性

**现象**:
- Qwen 模型有时以 XML 格式输出 tool_call 参数（非标准 JSON）
- Qwen 模型有时将 tool_call 作为 `<tools>` 嵌入 content text
- Rapid-MLX 忽略 `max_tokens`，可能生成无限制输出
- Qwen `reasoning_content` 字段需要特殊提取

**根因**: Claude Code 期望 Claude 模型的精确行为，但本地 Qwen 模型在工具调用格式、输出控制等方面存在差异。

**需求点**:

| ID | 需求 | 描述 | 优先级 |
|----|------|------|--------|
| R4.1 | **XML→JSON 回退** | 解析 Qwen 的非标准 tool_call 格式 | P0 |
| R4.2 | **Content-text 工具提取** | 从 `<tools>` 文本中提取工具调用 | P0 |
| R4.3 | **输出截断** | 代理层强制限制输出长度 | P0 |
| R4.4 | **JSON 修复** | 修复被截断的 tool_call arguments JSON | P1 |
| R4.5 | **Reasoning 提取** | 处理 Qwen 的 reasoning_content 字段 | P1 |

### 需求领域 5: 错误理解增强

**现象**:
- 后端返回英文技术错误（"Wasted call"、"File does not exist"）
- Qwen 模型对英文错误理解不如中文提示
- 模型反复对不存在的文件发起 Read

**根因**: 错误信息格式面向 Claude 设计，Qwen 对英文技术术语的理解和指令遵循度较低。

**需求点**:

| ID | 需求 | 描述 | 优先级 |
|----|------|------|--------|
| R5.1 | **错误翻译** | 将后端英文错误翻译为中文自然语言提示 | P1 |
| R5.2 | **错误上下文增强** | 翻译时附带解决建议（如"用 Bash ls 确认文件结构"） | P2 |

### 需求领域 6: 可观测性

**现象**:
- 代理处理过程复杂，包含 8 层管线 14+ 步骤
- 问题排查依赖自由文本日志，难以机器分析
- 缺少截断质量、压缩效果的量化指标

**根因**: 系统演进过程中功能逐步叠加，缺少统一的结构化指标体系。

**需求点**:

| ID | 需求 | 描述 | 优先级 |
|----|------|------|--------|
| R6.1 | **结构化 Metrics** | 每请求输出管线各步骤的结构化 JSON 指标 | P0 |
| R6.2 | **质量标记** | 自动标记异常模式（高丢弃率、压缩失败、循环检测） | P1 |
| R6.3 | **压缩比追踪** | 记录截断前后的 token 估算比 | P1 |

### 需求领域 7: 资源约束适配

**现象**:
- 两个并发大上下文请求 → `[METAL] Insufficient Memory` → 后端崩溃
- Rapid-MLX 的 `allocation_limit` 是软目标，峰值可超 20-40%
- Prefix cache 积累到 6GB+ 时风险剧增

**根因**: 48GB 统一内存需要被系统、模型权重、KV cache、prefix cache 共同分享，并发时内存不足。

**需求点**:

| ID | 需求 | 描述 | 优先级 |
|----|------|------|--------|
| R7.1 | **并发控制** | 通过 Semaphore 限制同时转发的请求数 | P0 |
| R7.2 | **云端切换** | 支持无缝切换到云端 API（DeepSeek/OpenAI），绕过本地内存限制 | P1 |

## 六、系统设计评估

### 6.1 架构优势

1. **完全代理层实现**: 所有优化在代理层完成，不需要修改 Claude Code 或推理引擎
2. **渐进式设计**: 功能按 Phase 迭代，每阶段独立可用
3. **配置驱动**: 25+ 环境变量控制，支持 local/cloud 模式自动适配
4. **零依赖**: 纯 Python stdlib，无第三方包，部署简单
5. **Prefix Cache 协同**: 代理层减 prompt + 引擎层缓存前缀 = 双层优化

### 6.2 架构风险

1. **管线复杂度**: 8 层 14 步骤的处理管线，维护成本高。`_handle_messages()` 函数体超过 400 行
2. **Token 估算不精确**: 使用 chars × ratio 估算 tokens，无 tiktoken，偏差可达 20-30%
3. **LLM 压缩单点**: `_compress_middle_with_llm()` 调用同一后端模型，增加延迟和资源竞争
4. **增量压缩缓存**: `_summary_cache` 纯内存，进程重启后丢失，首轮无法享受增量优势
5. **工具过滤的 Cache 权衡**: 过滤后工具列表变化导致 prefix cache miss，净收益依赖具体场景

### 6.3 待验证假设

| 假设 | 验证方式 | 风险 |
|------|---------|------|
| Level 2/3 能有效中断循环 | 收集 Metrics 中 loop_injected+level 数据 | 模型可能在 Level 2 后用 Bash 读取 → 新形式的循环 |
| Read 预览能降低重读率 | 对比有/无预览的 re_read_count | 预览可能不足 → 模型仍要 Read |
| 增量压缩信息保持质量 | 人工审查长会话的压缩摘要 | 多次合并后信息逐级丢失 |
| 工具过滤不影响任务完成率 | 对比有/无过滤的任务成功率 | 过滤掉的工具可能恰好是下一步需要的 |

## 七、下一步建议

基于当前 100% 需求覆盖率和已识别的潜在需求：

### 短期 (Phase 3: 调优)

1. **收集 Metrics 数据**: 让系统运行 1-2 周，收集 `proxy_metrics.jsonl` 数据
2. **验证关键假设**: 分析 loop_level 分布、re_read_count 趋势、compression_ratio 分布
3. **参数调优**: 根据数据调整 `PROXY_CTX_TOKEN_BUDGET`、`PROXY_LOOP_THRESHOLD`、`PROXY_TOOL_FILTER_MAX`

### 中期 (Phase 4: 增强)

4. **Re-read 硬拦截** (U5): 实现 Fix 1，在流式响应中拦截重复 Read
5. **自适应参数** (U4): 基于 Metrics 自动调整阈值
6. **BM25 Phase 2** (U1 前置): Bigram 分词 + 倒排索引，提升关键词匹配质量

### 长期 (探索)

7. **多模型协同** (U6): 9B 模型做压缩摘要，35B 做主推理
8. **跨会话记忆** (U1): JSONL 持久化 + 跨会话索引加载
9. **管线重构**: 将 `_handle_messages()` 拆分为独立函数或 pipeline 类，降低维护复杂度
